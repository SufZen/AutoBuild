AutoBuild — User Guide

Everything you need to go from zero to your first AI-built feature in under 10 minutes.

1. What is AutoBuild?

AutoBuild is a portable, markdown-based framework that turns any AI coding agent (Gemini, Claude, Cursor, Copilot, etc.) into a structured, self-improving software builder.

You define what you want built. The AI follows AutoBuild's structured workflow to plan, build, review, and iterate — producing higher-quality code than unstructured prompting.

Key concepts

Concept What It Means
Intent A description of what you want built (like a mini-spec)
Build program The strategy the AI follows (single-pass, multi-approach, autonomous loop)
Quality pipeline Seven automated checks that score every iteration 0–100
Sub-agents Specialist reviewers (security, architecture, tests, quality) that inspect the work
Learnings Knowledge accumulated from past builds that makes future builds better

AutoBuild is not an application. It's a set of markdown instructions. There's nothing to install, no dependencies, no runtime. You copy the files into your project and point your AI agent at them.

2. Installation

Prerequisites

  • A project with Git initialized
  • An AI coding agent (any will work)

Step-by-Step

1. Create the .autobuild/ directory in your project root.

2. Copy the workflow files from the AutoBuild repo:

Directory Layout 
.autobuild/
├── program.md              ← from core/program.md
├── quality-pipeline.md     ← from core/quality-pipeline.md
│
├── agents/                 ← entire agents/ folder
├── programs/               ← entire programs/ folder
├── intents/                ← entire intents/ folder
│
├── results/
│   ├── results.tsv         ← create with headers (see below)
│   └── learnings.md        ← from core/learnings-template.md
│
└── evaluation/             ← entire evaluation/ folder

3. Create results/results.tsv with this header row:

results.tsv 
timestamp	branch	intent	approach	quality_score	tests	lint	security	types	complexity	coverage	architecture	status	description
Important: In the AutoBuild repo, the source entry point is core/program.md. In your project, the installed entry point is .autobuild/program.md. They're the same file — just different locations.

3. Configuration

3a. Project Context

Copy setup/project-context.template.md → .autobuild/project-context.md and fill in project details. This file is your project's "constitution." The AI reads it before every task and follows its rules.

project-context.md 
## Project Overview
- **Name:** Acme API
- **Description:** REST API for the Acme e-commerce platform
- **Primary language:** Python 3.12
- **Framework:** FastAPI

## Anti-Patterns (NEVER DO)
1. Never use synchronous I/O in async contexts
2. Never import from internal modules of third-party packages

3b. Quality Config

Pick a stack preset from stacks/ (e.g., stacks/python.md) and copy it to .autobuild/quality-config.md.

Each preset maps onto AutoBuild's 7 canonical quality checks. Adjust weights to reflect your project's priorities (must total 100%).

Check What It Measures
TestsDo all tests pass?
Lint / FormatIs the code clean and formatted?
Type SafetyDo types check out?
SecurityAny vulnerabilities, secrets, or audit failures?
ComplexityIs the code reasonably simple?
CoverageHow much code is tested?
ArchitectureDoes it follow your project's conventions?

4. Writing Your First Intent

An intent is a small spec that tells the AI what to build. AutoBuild ships four intent templates: feature.md, fix.md, refactor.md, and optimize.md.

Quick Start

Copy _template.md to active-intent.md and fill it out:

active-intent.md 
## Goal
Add rate limiting to all public API endpoints.

## Scope
### IN
- Implement sliding-window rate limiter middleware
- Add tests for rate limiting behavior

### OUT
- Do not change authentication flow

## Build Mode
**Mode:** `standard`

5. Running a Build

Open your AI coding agent, point it at your project, and tell it:

"Read .autobuild/program.md and execute the AutoBuild workflow."

The agent will sequentially load context, analyze, plan, build, review, evaluate, and learn.

Iteration Logic

Code change → Run quality pipeline → Score 0-100 ↓ Score improved or held steady? No CRITICAL security issue? YES → Keep this state (git commit) NO → Discard (git reset) ↓ Is the score >= 80? → Ready to deliver Score 60-79? → Acceptable with notes Score < 60? → Not ready, keep iterating

6. Build Modes

Mode When to Use How It Works
Standard Simple, clear tasks Single-pass: implement → quality gate → review
Up to 3 fix-and-retry cycles if quality checks fail. Escalates to you if it fails.
Iterative Multiple valid approaches Try N approaches on branches, compare them, keep the best
Agent proposes 2-3 approaches automatically. The winner is polished through full review.
Overnight Large optimization space Autonomous loop until interrupted or plateaued
Never asks for input. Stops after 10 consecutive discards. Reverts failures immediately.
Optimize Target a specific metric Hill-climb toward a measurable goal (latency, size, etc.)
You define what to measure and the target. Stops when target reached or plateaued.

7. Understanding Quality Scores

Every iteration produces a composite score from 0 to 100 based on the seven weighted checks.

Two Decisions Per Score

AutoBuild makes two separate decisions from each score:

1. Branch Retention:

  • Improved/steady score + NO critical issues = Keep (commit)
  • Regressed score OR critical security issue = Discard (reset)

2. Delivery Readiness:

  • ≥ 80: Ready to deliver
  • 60 – 79: Acceptable with notes
  • < 60: Not ready (more work needed)

8. Reading Results

  • results/results.tsv — A complete history of every experiment the AI tried.
  • evaluation/latest-review.md — Generated at the end of each build containing the final score, sub-agent findings, and a human review checklist.
  • results/learnings.md — The growing knowledge base of what works for your project.

9. The Multi-Agent System

AutoBuild defines five specialist sub-agents. The orchestrator (program.md) activates them at the right time.

Agent When Activated What It Does
BuilderPhase 3Writes production-ready code following project conventions
Security ReviewerPhase 4Scans for secrets, CVEs, missing input validation
Architecture ReviewerPhase 4Checks compliance with architecture rules in project-context.md
Quality ScorerPhase 3/4Runs the quality pipeline and reports score, retention, and readiness
Test WriterPhase 4Strengthens test coverage for new/changed code

Any CRITICAL finding from Security or Architecture review sends the code back to the Builder for fixes before delivery.

10. Learnings & Self-Improvement

AutoBuild has a dual learning system. After each intent, it records what worked:

## 2026-03-25: Rate Limiting Implementation
- FastAPI middleware approach cleanest — decorator pattern caused import cycles
- Pattern: always test with burst requests, not just steady-state

Universal patterns can be placed in a globally referenced sharedlearnings.md.

11. Tips & Best Practices

Writing Good Intents

  • Be specific — "Add caching to the user endpoint" is better than "Make the app faster"
  • Define clear acceptance criteria — Testable, binary (done or not done)
  • Set explicit scope boundaries — OUT items prevent scope creep
  • Choose the right build mode — Don't use iterative for simple tasks

Git Workflow & Quality

  • AutoBuild always works on branches — never on the `main` branch.
  • Each kept improvement is a separate commit, easy to cherry-pick or revert.
  • Start with a quality preset, tune the weights after 3–5 builds.
  • After a tricky build, take 2 minutes to review and refine the auto-generated learnings in results/learnings.md.

12. Troubleshooting

  • "The agent doesn't follow the workflow": Make sure you tell it to read .autobuild/program.md specifically.
  • "The quality score is always 0": Check that your quality-config.md has the correct CLI commands for your stack. Verify they run locally first.
  • "The agent skips sub-agent reviews": Remind it to read Phase 4: SUB-AGENT REVIEWS in program.md.
  • "The agent modifies files outside the intent scope": Add an explicit anti-pattern in project-context.md: "Never modify files outside the intent's declared scope."
  • "Results aren't being logged": Verify the results.tsv file exists with the 14-column header row.
